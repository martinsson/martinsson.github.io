Test et refactoring de legacy

Améliorer (refactorer) du code existant sans risque de régression à l'aide d'écriture de tests unitaires, de l'utilisation de la couverture des tests et les refactorings automatisés de nos IDEs

Bénéfices pour les participants

    Voir comment rendre du code non testable testable. Connaître des principes pour bien refactorer. Apprendre comment les tests nous montrent des problèmes dans notre design. Être surpris par la capacité de refactoring automatique de nos IDEs.

TDD de la "vrai vie"

Démonstration comment on fait du TDD dans nos projets du quotidien, où l'on est confronté à des problèmes de testabilité (webservices tiers, dates, bases de données, ...)

Bénéfices pour les participants

    Observer et comprendre le cycle de TDD. Savoir comment tester (et concevoir) en présence de dépendances externes. Avoir un aperçu de la vraie vie du développeur TDD.

Golden Master / Approval testing

Comment faciliter le testing avec l'approche record-replay. On lance le test une fois pour "enregistrer" le fonctionnement de l'application. Ceci est le Golden Master. Lors des lancements suivants le résultat doit rester identique. Quelque part on automatise les assertions. A ceci près qu'on exploite l'intelligence humain lors de la validation du "Golden Master". Approvals tests est un autre nom de cette technique. Cette approche est souvent très utile lors du refactoring de legacy. Mais il a ses pièges.

Bénéfices pour les participants

    Connaitre cette technique et son outillage, arme redoutable notamment pour le legacy. Voir, à l'oeuvre les patterns liés à cette technique et savoir les pièges à éviter

Bug repellent, by design

Tous les styles de code ne se valent pas, même s'ils produisent le même résultat. Il est possible, de concevoir du code qui contiennent moins d'erreur que du code classique et qui est plus résistant à l'introduction de bugs dans le futur. Le code obtient une qualité intrinsèque complémentaire aux tests qui nous mêne plus loin que seul les tests pourraient le faire. Cette façon de concevoir est courant dans le design hors soft, mais nous avons la chance de manipuler une matière particulièrement peu cher à changer. Les opportunités sont donc nombreuses et facilement rentables. Au delà du design bas niveau, venez découvrir des idées pour votre équipe: l'entonnoir des erreurs, le typage vu autrement, comment ces idées survivent avec les microservices, comment la gestion du config et du déploiement changent. Couplage-cohésion, Connascence, if-less, No-Primitives, chasse aux cas limites.

Bénéfices pour les participants

    Comprendre que les bugs ne sont pas une fatalité. Savoir ce qui se cache derrière NoPrimitives. Pouvoir rendre les commentaires inutiles (souvent). Utiliser les types efficacement (même en JavaScript).
